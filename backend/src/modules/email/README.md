# Модуль email

Система приёма, обработки и хранения ведомостей из электронных писем через IMAP и SMTP. Модуль автоматически проверяет почтовый ящик, извлекает вложения (Excel-файлы), анализирует их структуру, сохраняет на диск и в базу данных, а также отправляет уведомления отправителю о результате обработки.

Инвентаризационные ведомости (определяются по ключевому слову "инвентар" в теме письма) не сохраняются в модуле email, а передаются в модуль inventory через EventEmitter.

## API Endpoints

**Защита**: JwtAuthGuard

### POST `/api/email/check`

Запускает ручную проверку почтового ящика. Подключается к IMAP, ищет непрочитанные письма, извлекает вложения. Обычные ведомости сохраняет в БД, инвентаризационные передаёт в inventory-модуль через EventEmitter.

**Доступ**: любая аутентифицированная роль.

**Ответ**:

    {
      "success": true,
      "message": "Проверка почты завершена"
    }

### GET `/api/email/attachments`

Возвращает список ведомостей, доступных текущему пользователю МОЛ. Фильтрация выполняется по складам из таблицы `mol_access`.

**Доступ**: роль МОЛ.

**Ответ**: массив EmailAttachmentResponseDto.

### DELETE `/api/email/attachments/:id`

Удаляет ведомость по ID. Проверяет существование записи, удаляет физический файл с диска и запись из БД. После успешного удаления отправляет SSE-уведомление `statementDeleted`.

Дополнительная проверка прав не требуется — пользователь может удалить только то, что видит в своём списке.

**Параметры пути**: `id` — числовой идентификатор ведомости.

**Доступ**: любая аутентифицированная роль.

**Ответ**:

    {
      "success": true,
      "message": "Ведомость удалена",
      "attachmentId": 5
    }


## Обработка вложений

### Поддерживаемые типы документов

**ОСВ** (оборотно-сальдовая ведомость)

Обязательные колонки: `Завод`, `Склад`, `КрТекстМатериала`, `Материал`, `Партия`, `Запас на конец периода`.

Параметры:
- `docType = 'ОСВ'`
- `zavod` — из колонки `Завод` (приводится к числу)
- `sklad` — из колонки `Склад` (первое непустое значение)

**ОС** (основные средства)

Обязательные колонки: `Основное средство`, `Название`, `Инвентарный номер`, `МОЛ`.

Параметры:
- `docType = 'ОС'`
- `zavod = 0`
- `sklad` — из колонки `МОЛ` (первое непустое значение)

### Два режима обработки

**Обычная ведомость** (тема письма не содержит "инвентар"):

1. Анализ Excel из Buffer (EmailFileAnalyzer)
2. Сохранение файла на диск (EmailStorageService)
3. Создание записи в email_attachments
4. SSE-уведомление `statementLoaded`
5. Email-уведомление отправителю о принятии

**Инвентаризационная ведомость** (тема письма содержит "инвентар"):

1. Анализ Excel из Buffer — проверка валидности (EmailFileAnalyzer)
2. Эмит события `inventory.file.received` с Buffer, именем файла и email отправителя (EventEmitter)
3. Файл не сохраняется на диск и в email_attachments
4. Дальнейшая обработка — в модуле inventory

### Алгоритм определения типа

1. Чтение Excel из Buffer через библиотеку xlsx
2. Проверка наличия листов и данных
3. Проверка первой строки на обязательные колонки ОСВ
4. Если не ОСВ — проверка на ОС
5. Если ни один тип не подходит — файл отклоняется

## Обработка ошибок

При ошибках анализа:
- Отправитель получает письмо с указанием причины
- Ошибка логируется в БД

При непредвиденных ошибках обработки:
- Уведомление отправляется администратору (tregubovsy@irkutsk-dobycha.gazprom.ru)
- Ошибка логируется в БД с полным стеком

## Управление ведомостями

### Права доступа

Доступ определяется через таблицу `mol_access`, которая связывает пользователя со складами (zavod, sklad). Метод `getAttachmentsForMol(userId)` возвращает только ведомости тех складов, к которым у пользователя есть доступ.

### Удаление

Удаление выполняется с проверкой:
- Ведомость существует (иначе `404`)
- Физический файл удаляется с диска
- Запись удаляется из БД
- Отправляется SSE-уведомление `statementDeleted`

### Конфигурация

Настройки загружаются через `ConfigModule` из переменных окружения.

## Интеграция с inventory-модулем

Модуль email не зависит от модуля inventory напрямую. Связь осуществляется через EventEmitter:

1. EmailProcessor эмитит событие `inventory.file.received` с данными:
   - `buffer` — содержимое Excel-файла
   - `filename` — оригинальное имя файла
   - `emailFrom` — адрес отправителя
   - `docType` — тип документа (ОСВ или ОС)

2. Inventory-модуль слушает это событие и парсит строки в таблицу `inventory_statements`.

3. EventEmitterModule.forRoot() должен быть в AppModule.


## Сервисы

### ImapService

Подключается к почтовому серверу по IMAP, ищет непрочитанные письма, извлекает вложения и передаёт их в EmailProcessor. После обработки помечает письма как прочитанные.

**Определение типа ведомости:**
Тема письма проверяется на вхождение "инвентар" (регистронезависимо). Результат передаётся в EmailProcessor.

### SmtpService

Отправляет email-уведомления через SMTP. Используется для информирования отправителя о результате обработки файла.

### EmailProcessor

Координирует обработку одного вложения:

- Принимает Buffer с содержимым файла
- Вызывает EmailFileAnalyzer для проверки валидности
- Для обычных ведомостей: сохраняет файл на диск, создаёт запись в БД, отправляет уведомление
- Для инвентаризационных: эмитит событие `inventory.file.received`, файл не сохраняет
- При ошибке: отправляет уведомление отправителю
- При непредвиденной ошибке: отправляет уведомление администратору

### EmailFileAnalyzer

Анализирует Excel-файл из Buffer. Не зависит от файловой системы.

**Возвращает:**
- `isValid` — валидность файла
- `docType` — тип документа ('ОСВ' или 'ОС')
- `zavod` — номер завода (из первой непустой строки)
- `sklad` — склад (из первой непустой строки)
- `error` — сообщение об ошибке (если невалидный)

### EmailAttachmentsService

Работа с записями в БД:

- `getAttachmentsForMol(userId)` — возвращает ведомости с фильтрацией по mol_access
- `deleteAttachment(id)` — удаляет ведомость, файл с диска, отправляет SSE

### EmailStorageService

Управляет физическими файлами на диске:

- `saveFile(filename, content)` — сохраняет файл с уникальным именем
- `deleteFile(filename)` — удаляет файл (не падает, если файл не найден)
- `fileExists(filename)` — проверяет существование
- `getFilePath(filename)` — возвращает полный путь

### Логирование

Модуль использует Logger от @nestjs/common. Основные события логируются:
- Подключение к IMAP и количество писем
- Сохранение и удаление файлов
- Результаты анализа (тип документа, склад)
- Передача инвентаризационных ведомостей в inventory-модуль
- Ошибки обработки

Глобальные перехватчики и фильтры логируют все HTTP-запросы и исключения в таблицу `logs`.
