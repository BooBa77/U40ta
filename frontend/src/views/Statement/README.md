# Модуль Statement — Страница ведомости МОЛ

## Обзор

Модуль `Statement` — страница для работы с ведомостью МОЛ, полученной из почтового вложения. Ведомости парсятся сразу при получении письма и сохраняются в БД. Отображает сгруппированные строки с цветовой индикацией, позволяет фильтровать записи, управлять актуальностью, создавать объекты через QR-сканирование и просматривать существующие объекты.

## Структура компонентов

### Основные компоненты

1. **StatementPage.vue** — корневой компонент, оркестрирует загрузку, фильтрацию и модалки
2. **StatementTable.vue** — таблица ведомости с группировкой, цветами и чекбоксами актуальности

### Модальные окна

3. **InvListModal.vue** — просмотр объектов и записей по инвентарному номеру и партии
4. **ObjectFormModal.vue** — создание/редактирование объекта (из модуля ObjectForm)
5. **UniversalFilterModal.vue** — универсальная модалка фильтрации (из `components/common`)

### Композаблы

- **useStatementData.js** — загрузка строк ведомости, вычисление `isExcess`, SSE-обновления
- **useStatementAggregation.js** — группировка строк, расчёт цветов и количества
- **useTableFilter.js** — логика фильтрации агрегированных данных (общий)
- **useActualManager.js** — управление флагом `isActual`

## Поток данных

0. **SSE**: при изменении объектов на складе другим МОЛ приходит событие `objects-changed`, вызывается `reload()`
1. **Загрузка**: `useStatementData` получает строки ведомости с бэка (с динамическим `objectCount`) или из IndexedDB
2. **isExcess**: после загрузки для каждого склада ведомости запрашиваются все объекты. Если объектов больше чем строк statements — добавляются виртуальные синие строки
3. **Агрегация**: `useStatementAggregation` группирует строки:
   - Синие (`isExcess = true`): группировка по `invNumber + partyNumber`
   - Активные (`isActual = true`): группировка по `invNumber + partyNumber`
   - Неактивные (`isActual = false`): группировка только по `invNumber`
4. **Цвета групп**:
   - Синий (0) — объекты есть в системе, но отсутствуют в ведомости (`isExcess`)
   - Красный (1) — все строки без объекта (`objectCount = 0`)
   - Жёлтый (2) — часть строк с объектом, часть без
   - Зелёный (3) — все строки с объектом (`totalCount = objectCount`)
   - Серый (4) — неактуальные (снят флаг `isActual`)
5. **Сортировка**: синие → красные → жёлтые → зелёные → серые, внутри группы по алфавиту
6. **Фильтрация**: `useTableFilter` фильтрует агрегированные строки по `invNumber` и `buhName`
7. **Отображение**: `StatementTable` отрисовывает отфильтрованные данные

## Функциональность

### 1. Просмотр ведомости

- Таблица с агрегированными строками и цветовой индикацией
- Отображение инвентарного номера, партии (для ОСВ), количества и наименования
- Синие строки — объекты не из ведомости, не имеют чекбокса и не кликабельны
- Клик по строке открывает `InvListModal` для просмотра объектов и записей

### 2. Фильтрация

- Фильтр по инвентарному номеру (`invNumber`)
- Фильтр по наименованию (`buhName`)
- Модалка `UniversalFilterModal` с поиском и множественным выбором
- Активные фильтры подсвечивают заголовки колонок жёлтым
- Кнопка «Сбросить фильтры» появляется при активных фильтрах

### 3. Управление актуальностью

- Чекбокс в каждой строке управляет флагом `isActual`
- Снятие флага переводит строку в неактуальные (серая группа, группировка без партии)
- Изменение применяется ко всем записям с данным `invNumber` в рамках ведомости
- Синие строки (`isExcess`) не имеют чекбокса

### 4. Создание объектов

- Через `InvListModal`: QR-сканирование для записей без объекта
- После создания объекта данные ведомости перезагружаются
- `haveObject` вычисляется динамически на бэке, не хранится в БД

### 5. Просмотр объектов

- `InvListModal` показывает два списка:
  - Записи ведомости без объекта (с кнопкой QR-сканирования)
  - Существующие объекты (с кнопкой «Открыть»)
- Фильтрация по `invNumber + partyNumber` для точного совпадения

### 6. Офлайн-режим

- Данные ведомости загружаются из IndexedDB
- `objectCount` вычисляется на фронте по данным из кэша
- Создание объектов работает офлайн (сохраняются в кэш)

## Интеграция с API

### Ведомость
- `GET /api/statements` — список ведомостей МОЛ
- `GET /api/statements/items?receivedAt=...` — строки ведомости (с динамическим `objectCount`)
- `DELETE /api/statements?receivedAt=...` — удаление ведомости
- `POST /api/statements/update-actual` — изменение `isActual`
- `GET /api/statements/by-inv?inv=&party=&zavod=&sklad=` — поиск записей

### Почта
- `POST /api/email/check` — проверка почтового ящика

### Объекты
- `GET /api/objects/by-inv?inv=&party=&zavod=&sklad=` — поиск существующих объектов
- `GET /api/objects/sklad?zavod=&sklad=` — все объекты на складе
- `POST /api/objects` — создание объекта
- `PATCH /api/objects/:id` — обновление объекта
- `GET /api/objects/place-combinations` — варианты местоположений

## Сервисы фронта

- `statement.service.js` — онлайн/офлайн работа с ведомостями
- `email.service.js` — проверка почтового ящика
- `object.service.js` — онлайн/офлайн работа с объектами
- `offline-cache-service.js` — кэширование в IndexedDB

## Стили

- Tailwind CSS на всех компонентах
- Цветовая схема групп через CSS-классы в `global.css`: синий/красный/жёлтый/зелёный/серый

## Меры безопасности

- JWT-авторизация на всех эндпоинтах
- Каждый МОЛ видит только свои ведомости (фильтрация по `user_id` из JWT)
- Проверка прав доступа к ведомости через `mol_access` на бэке
- Фронтовые проверки ролей через композабл `useCurrentUser`
